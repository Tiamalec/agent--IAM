# AI Agent IAM 原型开发计划

## 1. 目标重述
- 面向多 Agent 协作场景，先交付一个可运行、可验证的 **安全原型系统 MVP**。
- 优先覆盖比赛题目的四条主链路：**身份认证、细粒度授权、委托链、审计追溯**。
- 当前仓库尚无现成代码，因此第一阶段采用 **Python 标准库实现核心域能力**，避免过早绑定外部框架与依赖。

## 2. 第一阶段范围（本轮实现）
### In Scope
- Agent / User 注册与唯一身份标识
- HMAC 签名 Token（JWT-like）签发与校验
- RBAC + ABAC 混合授权校验
- 用户 -> Master Agent -> Worker Agent 的委托链
- Token 限时、限次使用
- 篡改可感知的 JSONL 审计日志
- 一个可直接运行的业务示例
- 覆盖核心主链路的单元测试

### Out of Scope
- 飞书 / OIDC 真正对接
- FastAPI / gRPC API 层
- 前端可视化界面
- Redis / PostgreSQL 持久化
- mTLS 真证书体系

## 3. 技术路线
### 阶段 A：核心安全内核
1. 设计统一数据模型：Actor、Policy、TokenClaims、AuditEvent。
2. 实现签名 Token 服务，支持：
   - 主题主体（subject）
   - 签发方（issuer）
   - scopes
   - parent token
   - trust chain
   - 过期时间
   - 使用次数限制
3. 实现授权引擎：
   - RBAC：角色与动作匹配
   - ABAC：主体属性 / 请求上下文属性 / 时间窗校验
4. 实现委托逻辑：
   - 父 token 才能派生子 token
   - 子 token scope 必须是父 token 子集
   - 继承并扩展 trust chain
5. 实现审计：
   - 每次签发 / 授权 / 拒绝 / 心跳写入日志
   - 以前序哈希串联，提升防篡改可检测性

### 阶段 B：接入层
1. 暴露 REST API（建议 FastAPI）
2. 对接真实身份源 / 登录授权流
3. 接入数据库与缓存

### 阶段 C：演示层
1. 多 Agent 协作 Demo
2. 审计链路可视化
3. 异常行为拦截演示（如超权、超次、过期、上下文不匹配）

## 4. 里程碑
### M1：安全内核 MVP（当前开始）
- 产出可运行 Python 模块
- 可模拟“财务只读委托成功 / 越权写入失败”
- 测试通过

### M2：接口化
- 增加 HTTP API
- 输出 OpenAPI 文档

### M3：可视化与比赛演示
- 接入前端页面
- 展示委托链和审计图

## 5. 验收标准
- 能注册 User / Master Agent / Worker Agent
- 能签发用户授权给 Master 的根 token
- 能由 Master 派生只读 Worker token
- Worker 读取财务报表成功
- Worker 写入财务报表被拒绝
- 超过 max_uses 的 token 被拒绝
- 篡改 token 可被检测
- 审计日志记录 allow / deny / delegate / issue 等关键事件

## 6. 当前代码落地计划
1. 创建 `agent_iam/` 包，实现核心域模型与服务。
2. 创建 `tests/`，锁定核心行为。
3. 增加 `agent_iam/demo.py`，用于比赛演示与联调。
4. 后续再接 API 层，避免过早引入第三方依赖。

## 7. 风险与应对
- **风险：** 现在没有真实身份源。  
  **应对：** 先抽象 token/subject/issuer 关系，后续可替换为 OIDC。

- **风险：** 无数据库时日志不可真正防删改。  
  **应对：** 先实现 hash-chain 审计原型，后续替换为数据库或 WORM 存储。

- **风险：** ABAC 规则可能不断扩展。  
  **应对：** 先用结构化 Policy 模型，后续再扩成 DSL 或策略引擎。
